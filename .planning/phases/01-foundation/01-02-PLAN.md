<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| App → Database | All DB access flows through the Drizzle ORM, which parameterizes queries — the boundary is the db client singleton in src/lib/db/index.ts |
| Migration files → Database | Generated SQL is applied by the migrator; the migration folder is the trusted source — any drift means an unapplied schema change reached production silently |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-01-05 | Tampering | SQL injection via raw queries | high | mitigate | Drizzle ORM parameterizes all queries by default (RESEARCH.md Security Domain). The client singleton in src/lib/db/index.ts is the only DB entry point. No raw string-concatenated SQL anywhere. |
| T-01-06 | Tampering | Migration drift (schema.ts edited, migration not regenerated) | high | mitigate | Two-layer drift gate per RESEARCH.md Pitfall 1: (1) pnpm test:migrations clean-room test fails if migrations do not apply cleanly; (2) drizzle-kit generate produces new files if schema drifted (caught by git diff --exit-code in CI). drizzle-kit check is snapshot-consistency-only and NOT relied on as the drift gate. |
| T-01-07 | Information Disclosure | DATABASE_URL in env | medium | mitigate | DATABASE_URL lives only in gitignored .env.local (Plan 03 ships .env.example with the local default, but real secrets stay in .env.local). Never logged via lib/log. Never prefixed NEXT_PUBLIC_. |
| T-01-08 | Repudiation | Missing audit trail on schema changes | low | accept | Migration files ARE the audit trail (forward-only, committed to git). Phase 7 adds backup-restore drill cadence. No additional audit table needed in Phase 1. |
</threat_model>
