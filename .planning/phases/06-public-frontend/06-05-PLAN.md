src/actions/contact-schema.ts — pure Zod schema module (NO "use server" directive — "use server" files can only export async functions, so the schema MUST live here per the established pattern from seo-settings-schema.ts). Define `contactSchema = z.object({ name: z.string().min(1), email: z.string().email(), subject: z.string().optional(), message: z.string().min(1), website: z.string().optional() })` — the "website" field is the honeypot (bots auto-fill hidden fields named "website"/"url"). Export `type ContactInput = z.infer<typeof contactSchema>`. Re-export `{ zodResolver } from "@hookform/resolvers/zod"` so the client form imports both from one module.

src/actions/contact.ts — "use server" at the top. Export `submitContact(input: unknown): Promise<{ ok: true }>`. NO 'use cache' (Pitfall 7 — Server Actions are mutations, never cached). NO requireCan (contact is unauthenticated — the only controls are honeypot + rate-limit per D-07).

Implementation order inside submitContact:
1. Parse input via contactSchema.parse(input) — throws on invalid input (Zod validation).
2. Honeypot check: if data.website is non-empty, return { ok: true } immediately WITHOUT sending (silent success — the bot thinks it worked; D-07).
3. Rate-limit: extract IP from the x-forwarded-for header via headers() from next/headers. Call tryConsume(ip, 5, 3600000) — 5 submissions per IP per hour (CONTEXT.md discretion item). If false, throw new Error("RATE_LIMITED").
4. Read recipient: call getSetting("contact.recipient_email") from @/actions/settings. If empty/null, fall back to "admin@anydiscussion.com" (dev default — the admin sets the real address via the dashboard settings).
5. Send email: call sendEmail({ to: recipient, subject: `Contact: ${data.subject ?? data.name}`, text: `From: ${data.name} <${data.email}>\n\n${data.message}` }). sendEmail is fire-and-forget — it NEVER throws (Phase 2 R8), so a send failure does not propagate to the user.
6. Return { ok: true }.
  </action>
