## Phase Goal (Gap-Closure Scope)

**I want to** request a password reset from the signin page and complete the reset via the emailed link,
**so that** I can regain access to my account without an admin's help.

The server-side hook (`sendResetPassword` at `src/lib/auth/index.ts:55`) IS wired and its firing is proven by `__tests__/email-flows.test.ts`. This plan delivers the missing UI that triggers it and the page that lands the reset link.

## API Verification (from node_modules/better-auth@1.6.23)

**The UAT gap referenced `authClient.forgetPassword` — that method DOES NOT EXIST in Better Auth 1.6.23 core.** Verified by reading `node_modules/better-auth/dist/client/proxy.mjs` (uses `toKebabCase` reverse-mapping from property chain to endpoint path) and `node_modules/better-auth/dist/api/routes/password.mjs`:

| Client method (verified) | Endpoint | Body | Returns |
|---|---|---|---|
| `authClient.requestPasswordReset({ email, redirectTo })` | POST `/request-password-reset` | `{ email: string, redirectTo?: string }` | `{ status: boolean, message: string }` |
| `authClient.resetPassword({ newPassword, token })` | POST `/reset-password` | `{ newPassword: string, token?: string }` | `{ status: boolean }` |

**Email link flow (verified from `password.mjs` lines 71-72, 118):**
- Better Auth generates: `${baseURL}/reset-password/${token}?callbackURL=${encodeURIComponent(redirectTo)}`
- Clicking the link hits GET `/reset-password/:token` → server verifies token → redirects to `${redirectTo}?token=${token}`
- So passing `redirectTo: '/reset-password'` makes the user land on `/reset-password?token=xxx`

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → /api/auth/[...all] (request-password-reset) | Email crosses here; Better Auth generates token + fires sendResetPassword hook |
| Browser → /api/auth/[...all] (reset-password) | New password + token cross here; Better Auth validates token + hashes password |
| Email link → /reset-password?token=xxx | Untrusted URL parameter (token) crosses here; validated server-side by Better Auth |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-02-04 | Information disclosure | forgot-password response reveals whether email exists | medium | mitigate | ForgotPasswordForm always shows "check your email" regardless of success/error — email-enumeration protection (Better Auth also returns the same `{ status: true, message: "If this email exists..." }` for both cases server-side) |
| T-02-10 | Tampering | forged/expired token submitted to reset-password | medium | mitigate | Better Auth's resetPassword endpoint validates the token against the verification table server-side (consumeVerificationValue) — expired/forged tokens are rejected; the form displays the error |
| T-02-11 | Spoofing | open redirect via the redirectTo param | low | accept | redirectTo is hardcoded to "/reset-password" in the form code — it is never user-controlled; Better Auth's originCheck middleware validates the callbackURL on the GET callback endpoint |
| T-02-SC | Tampering | npm installs | high | mitigate | No new npm packages this plan (uses existing better-auth, @testing-library/react). No package legitimacy gate needed. |
</threat_model>
