## Phase Goal

**As a** first-time operator of a freshly-deployed instance,
**I want to** create the initial admin account via a setup screen that self-disables, then sign in with those credentials,
**so that** the dashboard is bootstrapped securely and no open privilege-escalation endpoint remains.

This plan delivers the sign-in slice + the first-run admin setup slice: the auth browser client, the `createFirstAdmin` Server Action with the D-08 self-disable, the repurposed signup page, and the signin form wired to Better Auth.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → /api/auth/[...all] (signin) | Credentials cross here; Better Auth verifies + sets httpOnly cookie |
| Browser → createFirstAdmin action | The bootstrap boundary — MUST be closed after the first admin (D-08) |
| Browser → createUser/banUser actions | Admin-only mutations; each re-checks requireCan server-side |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-02-02 | Elevation of privilege | createFirstAdmin (open setup endpoint) | critical | mitigate | D-08: count(admins)===0 server-side check BEFORE any auth.api call; refuses with FORBIDDEN after first admin; test "createFirstAdmin blocked" proves it (R1) |
| T-02-07 | Information disclosure | signin error reveals valid emails | medium | mitigate | requireEmailVerification:true (Plan 02-01) + Better Auth generic error responses; no per-email messaging in the SignInForm error UI |
| T-02-08 | Spoofing | forged callbackURL redirects to malicious host | medium | mitigate | callbackURL falls back to /dashboard; the `next` param is only ever set by proxy.ts from the original pathname (same-origin) — do not accept absolute URLs |
| T-02-09 | Tampering/Elevation | createUser/banUser called without admin permission | high | mitigate | Each action starts with requireCan({user:['create'/'ban']}) — Pitfall #1; the proxy gate is not trusted (Pitfall #4) |
| T-02-SC | Tampering | npm installs | high | mitigate | No new npm packages this plan (better-auth/resend installed in 02-01). drizzle-orm stays ^0.45.2 (R5). |
</threat_model>
