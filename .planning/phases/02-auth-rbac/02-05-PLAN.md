<objective>
Close the Phase 2 UAT Test 3 BLOCKER: an unauthenticated browser visiting /dashboard renders the dashboard instead of redirecting to /signin (AUTH-03 broken at runtime).

Root cause (diagnosed in .planning/debug/dashboard-auth-gate-bypass.md): proxy.ts is compiled by Turbopack but never registered in middleware-manifest.json (empty `middleware: {}` in both dev and prod), so Next.js routes zero requests through the proxy. Compounding this: the (admin) layout and dashboard page have NO server-side getSession() check, and the page is statically prerendered under cacheComponents:true, so it serves to all users. The existing __tests__/proxy.test.ts called proxy(req) directly with mocked cookies — validating function logic but never that Next routes real HTTP through the proxy (false confidence; 24 tests green).

Purpose: Restore the auth boundary with TWO layers. Layer 1 (authoritative): a server-side getSession() check in the (admin) route-group layout that redirects unauthenticated users to /signin — this is the real RBAC boundary the UX-only proxy was never meant to be (Pitfall #4). Layer 2 (UX): resolve the proxy manifest-registration gap so the cookie-existence gate also fires. Layer 3 (test): an integration test that sends a real no-cookie HTTP request to /dashboard and asserts the redirect — the class of test that was missing and let this bug ship.

Output: A Server-Component (admin) layout with a getSession() gate, an extracted client shell, a verified proxy/middleware registration, and a regression integration test.
</objective>

    - Behavior 1: When getSession() returns null (no session cookie / expired / forged), the (admin) layout redirects to /signin and renders NO dashboard shell or content.

  </behavior>
  <action>
    This is the AUTHORITATIVE fix — it closes the gap even if the proxy layer stays completely broken. The proxy was always documented UX-only (Pitfall #4); a server-side session check must be the real boundary (project CLAUDE.md: "Permission checks are never optional").

    STEP 1 — Extract the client shell. Create src/app/(admin)/AdminShell.tsx as a client component (it must use the useSidebar hook, which is a client-only context). Move the ENTIRE body of the current src/app/(admin)/layout.tsx into AdminShell.tsx verbatim: the "use client" directive, the useSidebar() destructure, the mainContentMargin computation, and the JSX that renders AppSidebar, Backdrop, AppHeader, and the children container. AdminShell's default export takes { children } and renders the same markup. Do NOT change any className, any structure, or any import — this is a pure relocation so the layout itself can become a Server Component. The SidebarProvider lives in the root layout (src/app/layout.tsx, above this route group), so useSidebar() inside AdminShell still resolves correctly when AdminShell is rendered as a child of the new server layout.

    STEP 2 — Convert the layout to a Server Component. Rewrite src/app/(admin)/layout.tsx so it has NO "use client" directive. It becomes an async Server Component that: (a) calls getSession() imported from "@/lib/auth/server" (the existing barrel that re-exports getSession from the Better Auth instance — src/lib/auth/server.ts already exports it); (b) if the returned session is falsy, calls redirect("/signin") imported from "next/navigation" and returns nothing (redirect throws at the Next.js runtime level, short-circuiting the render — no further JSX executes); (c) otherwise returns <AdminShell>{children}</AdminShell>. Export the layout as `export default async function AdminLayout({ children }: { children: React.ReactNode })`. Do not import useSidebar, AppHeader, AppSidebar, or Backdrop into the layout anymore — those now live in AdminShell. Import only getSession and redirect (and React for the ReactNode type if needed). Do NOT add metadata here (metadata stays on the page, not the layout).

    STEP 3 — Why this makes the route dynamic. getSession() calls auth.api.getSession({ headers: await headers() }). The headers() call from next/headers is a dynamic API in Next.js 16; using it opts the (admin) route group out of static prerendering. This means /dashboard is no longer marked ○ (Static) at build time — it becomes ƒ (Dynamic), evaluated per-request, so the redirect fires for each unauthenticated request and no static HTML is cached and served to logged-out users. Do not add force-dynamic or any config flag — the headers() usage alone is sufficient and idiomatic.

    Reference: getSession signature is `export async function getSession()` returning `{ user, session } | null` (src/lib/auth/index.ts lines 107-109). redirect is from "next/navigation" (Next 16 — works in Server Component layouts, throws a redirect response). The existing 02-01 SUMMARY documents the proxy as "UX-ONLY — NOT authoritative RBAC (Pitfall #4)" — Task 1 here installs the authoritative boundary that was missing.
  </action>
