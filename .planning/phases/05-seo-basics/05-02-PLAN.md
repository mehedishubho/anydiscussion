<objective>
Plan 02 — Standalone SEO Routes: sitemap.ts + robots.ts + rss.xml (Wave 2, parallel with Plan 03)

This plan delivers three standalone root routes that make the blog crawlable and subscribable: a dynamic `app/sitemap.ts` listing every published post + page with per-content-type priority/changefreq, a `app/robots.ts` with the allow/disallow rules + sitemap pointer, and an `app/rss.xml/route.ts` Route Handler emitting a full-text RSS 2.0 feed of the latest published posts. After this plan, `/sitemap.xml`, `/robots.txt`, and `/rss.xml` all return correct content, and they refresh without a full rebuild because the publish action already calls `revalidatePath` on these paths (D-13 carry-forward).

Purpose: Search engines and feed readers can discover and consume the blog. The sitemap is the single most important SEO artifact after per-page metadata; robots.txt gates crawler access away from dashboard/preview routes; RSS is the reader-maximum-reach posture (PROJECT.md core value — full-text, not excerpt-only). All three read from the `getSeoSettings()` cached snapshot (Plan 01) so the canonical base URL is consistent everywhere (Pitfall 7).

Output: `src/app/sitemap.ts`, `src/app/robots.ts`, `src/app/rss.xml/route.ts`, and three Wave 0 unit-test files (with DB mocked for the route tests).

Decisions implemented: D-05 (sitemap posts+pages, single, priority/changefreq), D-06 (robots disallow list), D-07 (RSS full-text posts-only, /rss.xml).
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| posts.body (Tiptap JSON) → renderPostBody → RSS <content:encoded> | Editor-authored HTML crosses into a public XML feed; must be sanitized to prevent stored XSS in feed readers |
| post.title/excerpt → escapeXml → RSS/sitemap text nodes | User-authored strings render into XML text fields; must be escaped to prevent XML injection/breakage |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-05-02 | Tampering / Spoofing | RSS <content:encoded> body (stored XSS via feed reader) | high | mitigate | renderPostBody runs the Phase 3 CONT-03 double-sanitize pipeline (generateHTML → sanitizeBeforeRender) BEFORE the body enters the XML; CDATA wrapping is defense-in-depth, NOT a substitute. Verified at src/lib/post-render.ts lines 39-45. |
| T-05-04 | Tampering | XML injection via post title/excerpt in RSS and sitemap URL fields | medium | mitigate | escapeXml on all text fields (the five XML-special characters). Sitemap URL fields use only Latin slugs (Phase 3 D-20 regex-validated) so URL injection is not a surface. |
| T-05-05 | Information Disclosure | draft / soft-deleted posts leaking into sitemap or RSS | medium | mitigate | Both queries filter status equals "published" AND deletedAt is null — verified by unit test (draft + soft-deleted fixtures excluded). Preview tokens are also implicitly excluded (they are not in the published set). |
| T-05-SC | Tampering | npm installs | low | accept | Zero packages installed this phase (all primitives ship with next@16.2.9). No slopsquat surface. |
</threat_model>
